وعليكم السلام ورحمة الله وبركاته،
هذا تحليلٌ أوليٌّ احترافيٌّ ومتكاملٌ لنظام توصيل وأمانات قابل للتوسع على مستوى جميع المحافظات. بنيته هنا كوثيقة قرار قبل التنفيذ، بحيث تُراجعها ثم ننتقل بعدها إلى التصميم التفصيلي وقاعدة البيانات والشاشات النهائية.

1) فكرة النظام بالكامل

النظام ليس “تسجيل شحنات” فقط، بل هو منصة تشغيل ولوجستيات ومالية في الوقت نفسه. أفضل تصور له هو أنه يدير 4 مسارات معًا:

مسار الشحنة، 2) مسار المال، 3) مسار الحركة بين الفروع والوكلاء، 4) مسار الرقابة والتدقيق.

دورة العمل المقترحة

استلام الشحنة من العميل في الفرع أو عبر وكيل أو عبر طلب إلكتروني.

إنشاء الشحنة كمسودة ثم اعتمادها بعد إدخال البيانات الأساسية.

حساب السعر تلقائيًا أو يدويًا مع تسجيل سبب أي تعديل.

إصدار رقم تتبع فريد وطباعة ملصق باركود/QR.

تسليم الشحنة للمخزن/الفرز مع تسجيل المستلم والوقت والموقع.

الفرز الداخلي حسب المحافظة/المنطقة/الفرع/الوكيل.

النقل بين المحافظات عبر سائق أو خط نقل أو شحنة مجمعة.

استلامها في الفرع المستهدف مع مطابقة عدد القطع والأوزان.

تسليمها للوكيل الخارجي أو موظف التوصيل.

محاولة التسليم للعميل النهائي.

إثبات التسليم بتوقيع/صورة/رمز تحقق/موقع GPS إن لزم.

إغلاق الشحنة ماليًا إذا كانت عليها مبالغ تحصيل أو رسوم مؤجلة.

تسوية العمولة والمستحقات للوكيل أو السائق أو الفرع.

التعامل مع الأمانات والطرود

الأمانة: شيء ذو قيمة أو مهمة يجب تتبعه بدقة وقد يتضمن تحصيل/تسليم/إرجاع.

الطرد: شحنة مادية قد تحتوي عدة عناصر.

الأفضل أن يكون كل سجل شحنة قادرًا على احتواء: 

قطعة واحدة أو عدة قطع.

قيمة مُصرح بها.

رسوم شحن.

رسوم تحصيل.

خدمة إضافية.

حالة مالية منفصلة عن حالة النقل.

تتبع الحالة

كل شحنة يجب أن تمتلك:

حالة رئيسية مثل: مسجلة، مستلمة، في الفرز، في النقل، وصلت الفرع، مع الوكيل، خرجت للتسليم، تم التسليم، فشل التسليم، مرتجعة، مغلقة.

سجل أحداث كامل لكل تغيير حالة، مع: 

من قام بالعملية

متى تمت

أين تمت

من سلّم ومن استلم

ملاحظات وصور إن وجدت

هذا السجل أهم من الحالة نفسها، لأنه هو الذي يعطي الشفافية والمساءلة.

التوسع المستقبلي

اجعل بعض الحقول المرنة في jsonb لتخزين بيانات إضافية حسب الحاجة. PostgreSQL يدعم jsonb كما يدعم الفهارس والتقسيم والـ partial indexes، وهذا مفيد جدًا إذا أردت توسعًا في بيانات الشحن أو الاستثناءات أو خصائص الطرود لاحقًا.

11) اختيار التقنيات

الواجهة الأمامية

Next.js + React
React مبني على المكوّنات، وNext.js App Router يعتمد على React الحديثة مثل Server Components وSuspense وServer Functions، مع تقسيم تلقائي للملفات لتخفيف التحميل الأولي. هذا مناسب جدًا للوحة الإدارة، وللعميل، وللشاشة الجوال الخاصة بالوكيل. 

الخلفية

ASP.NET Core
السبب:

موثق رسميًا كإطار عمل عالي الأداء وآمن ومتعدد المنصات ومناسب للتطبيقات السحابية. 

لديه Identity مدمج لإدارة المستخدمين والكلمات المرور والأدوار والـ claims والـ tokens. 

يفصل بوضوح بين authentication وauthorization. 

مناسب جدًا للتطبيقات التي تحتاج تحديثات لحظية مثل التتبع، مع SignalR. 

بديل قوي

NestJS

NestJS إطار Node.js منظم ومبني على TypeScript، ويعتمد على modules وproviders لتنظيم التطبيق. 

ممتاز إذا كان فريقك مائلًا إلى TypeScript بالكامل.

لكنه في مشروع مالي/تشغيلي كبير جدًا، أميل إلى ASP.NET Core من ناحية النضج المؤسسي والهوية والأمان والأداء.

Laravel

Laravel قوي جدًا في الإنتاجية السريعة، ويقدّم queues وvalidation وبث الأحداث بشكل منظم. 

لكنه أقل ملاءمة من ASP.NET Core إذا كان الهدف هو منصة تشغيل كبيرة جدًا مع real-time كثيف وضبط مؤسسي طويل الأمد.
